# 随手记导入功能设计

日期：2026-05-06

## 1. 背景和目标

BillBoard 当前的核心体验是两人家庭快速记账、按成员视角查看、按时间范围复盘。新增导入功能的目标是把已有记账软件中的历史数据迁移进同一套家庭账本，避免用户重新手工录入大量历史记录。

第一期只支持随手记 `.xlsx` 导入。后续可以继续扩展支付宝账单、微信账单等来源，但一期不实现这些来源。

已确认的产品目标：

- 从 `记录` 页进入导入流程。
- 支持随手记导出的 `支出` 和 `收入` 工作表。
- 导入时由用户指定随手记分类到 BillBoard 现有分类的映射。
- 不自动新增 BillBoard 分类。
- 同一家庭共享同来源的分类映射记忆，下次导入自动复用。
- 同源同指纹记录自动跳过，避免重复导入。
- 跨来源或手动记录中出现同日期同金额的记录时，集中提示为疑似重复，默认都保留，由用户改少量例外。
- 上传后生成导入草稿，不长期保存原始 Excel 文件。

样例文件观察：

- 文件名：`随手记Lehary-Noma的家20260506.xlsx`
- 工作表：`支出`、`收入`
- `支出` 约 9072 条记录，`收入` 约 253 条记录。
- 两张表列结构一致，包含 `交易类型`、`日期`、`一级分类`、`二级分类`、`账户币种`、`金额`、`成员`、`记账人`、`备注` 等字段。

## 2. 已确认假设

- 一期只接受随手记 `.xlsx` 文件。
- 所有导入数据都写入当前登录用户所在的 `householdId`。
- 金额继续使用整数分 `amountFen`。
- 日期和范围语义继续按 `Asia/Shanghai` 处理。
- 有效记录查询继续排除软删除数据。
- 导入功能必须校验当前 session 的 `householdId` 和 `memberId`。
- `李环宇` 和 `老公` 映射到 BillBoard 的“老公”成员。
- `晶晶` 和 `老婆` 映射到 BillBoard 的“老婆”成员。
- 随手记 `成员` 用作交易归属人；为空时回退到 `记账人`。
- 随手记 `记账人` 用作 BillBoard 的 `createdByMemberId`；无法映射时回退到当前登录用户。
- 如果交易归属人也无法映射，同样回退到当前登录用户，并在预览中提示。

## 3. 方案选择

推荐并采用方案：导入草稿 + 来源适配器 + 家庭共享映射记忆。

### 3.1 采用方案

上传文件后，服务器解析随手记工作簿，生成数据库中的导入草稿。草稿保存规范化后的行、映射状态、同源重复状态、疑似重复状态和用户选择，不长期保存原始 Excel 文件。用户补全映射并确认疑似重复后，系统批量写入 `Transaction`。

选择理由：

- 适合 9000 条以上历史记录。
- 上传、映射、重复确认、最终写入可以分步完成。
- 页面刷新后仍可继续处理当前草稿。
- 后续扩展支付宝、微信时，只需要增加来源适配器和字段映射。
- 分类映射记忆、同源去重、跨来源疑似重复确认都有明确落点。

### 3.2 备选方案

浏览器临时预览：数据库模型更少，但大文件和长流程不稳定，刷新会丢失状态，不适合历史迁移。

后台自动导入：操作更少，但不满足用户指定分类映射和疑似重复集中确认的要求，容易污染账本。

## 4. 数据模型设计

### 4.1 `Transaction` 导入来源字段

在现有 `Transaction` 表增加导入来源元数据：

- `source`：可为空，导入来源。例如 `sui_shou_ji`。
- `sourceFingerprint`：可为空，同源去重指纹。
- `sourceImportedAt`：可为空，导入写入时间。

同一家庭下，`source + sourceFingerprint` 一致时视为同源精确重复，导入草稿行自动标记为跳过。

手动记录的 `source` 和 `sourceFingerprint` 为空。跨来源或手动疑似重复不能靠这些字段自动跳过，只进入人工确认。

### 4.2 `ImportCategoryMapping`

新增家庭共享的分类映射表。核心字段：

- `id`
- `householdId`
- `source`
- `transactionType`
- `primaryCategory`
- `secondaryCategory`
- `categoryId`
- `createdAt`
- `updatedAt`

唯一约束：

- `householdId + source + transactionType + primaryCategory + secondaryCategory`

语义：

- 映射按家庭共享，不按登录用户隔离。
- 映射目标只能是现有 BillBoard 分类。
- 映射目标分类类型必须匹配交易类型。
- 导入确认时保存新映射；下次同来源导入自动复用。
- 出现新的 `交易类型 + 一级分类 + 二级分类` 组合时，用户必须补充映射后才能确认导入。

### 4.3 `ImportDraft` 和草稿行

新增导入草稿模型，用于承载一次上传后的解析结果和确认状态。

`ImportDraft` 核心字段：

- `id`
- `householdId`
- `source`
- `status`：`PENDING`、`COMPLETED`
- `fileName`
- `createdByMemberId`
- `createdAt`
- `updatedAt`
- `completedAt`

草稿行可以用独立表保存，核心字段：

- `id`
- `draftId`
- `rowNumber`
- `transactionType`
- `occurredAt`
- `amountFen`
- `actorMemberId`
- `createdByMemberId`
- `mappingKey`
- `categoryId`
- `note`
- `sourceFingerprint`
- `status`
- `skipReason`
- `duplicateCandidates`
- `userDecision`

草稿行状态建议覆盖：

- `READY`：可导入。
- `NEEDS_MAPPING`：缺分类映射。
- `SOURCE_DUPLICATE`：同源同指纹，自动跳过。
- `POSSIBLE_DUPLICATE`：跨来源或手动同日期同金额，默认保留。
- `INVALID`：行数据无法导入。
- `USER_SKIPPED`：用户选择跳过。

## 5. 随手记解析规则

### 5.1 工作表和表头

只读取 `支出` 和 `收入` 两张表。每张表必须包含：

- `交易类型`
- `日期`
- `一级分类`
- `二级分类`
- `账户币种`
- `金额`
- `成员`
- `记账人`
- `备注`

可选读取：

- `账户1`
- `商家`

如果表头缺失关键列，上传失败，不生成草稿。

上传 action 在读取文件内容前限制 `.xlsx` 文件不超过 20MB。来源解析器使用 `exceljs` 读取工作簿，并限制 `支出`、`收入` 两张表合计可解析数据行不超过 20000 行；超限时不生成草稿，显示文件过大错误。

### 5.2 字段转换

- `交易类型`：`支出` 映射为 `EXPENSE`，`收入` 映射为 `INCOME`。
- `账户币种`：只接受 `CNY`，其它币种标记为不可导入。
- `金额`：转为整数分，必须大于 0。
- `日期`：按 `Asia/Shanghai` 解析为 `occurredAt`。
- `一级分类`、`二级分类`：去除首尾空白，用于映射 key。

### 5.3 成员映射

交易归属人：

1. 优先读取 `成员`。
2. `成员` 为空时读取 `记账人`。
3. `李环宇/老公` 映射为“老公”。
4. `晶晶/老婆` 映射为“老婆”。
5. 无法映射时回退为当前登录用户，并在预览中提示。

创建人：

1. 读取 `记账人`。
2. `李环宇/老公` 映射为“老公”。
3. `晶晶/老婆` 映射为“老婆”。
4. 无法映射时回退为当前登录用户。

### 5.4 备注生成

导入后写入 BillBoard `Transaction.note`。备注保留：

- 原始 `备注`
- 随手记 `一级分类 / 二级分类`
- `成员`
- `记账人`
- `账户1`
- `商家`

不把完整原始行塞入备注，避免历史记录过于臃肿。

推荐格式：

```text
原始备注
随手记：一级分类 / 二级分类；成员：成员；记账人：记账人；账户：账户1；商家：商家
```

没有原始备注时只写随手记补充信息。

### 5.5 来源指纹

来源指纹用于同源精确去重。随手记一期基于稳定原始字段生成：

- 来源
- 交易类型
- 日期
- 金额
- 成员
- 记账人
- 一级分类
- 二级分类
- 账户1
- 商家
- 账户币种
- 原始备注

同一个家庭下已存在同 `source + sourceFingerprint` 的交易时，草稿行标记为 `SOURCE_DUPLICATE`，自动跳过。

## 6. 重复处理设计

### 6.1 同源精确重复

同源精确重复自动跳过，不要求用户逐条确认。预览页显示统计数量。

判断规则：

- 当前草稿行有 `source` 和 `sourceFingerprint`。
- 当前家庭中已有非软删除或已软删除交易的同 `source + sourceFingerprint` 记录。

同源去重包含软删除记录，避免用户删除后再次导入把同一历史记录恢复回来。

### 6.2 跨来源或手动疑似重复

疑似重复规则：

- 同一家庭。
- 同一天。
- 同金额。
- 来源不同，或已有记录为手动记录。

这只是弱匹配，不自动跳过。默认决策是“保留导入”。预览页集中列出疑似重复记录和候选已有记录，用户可以把个别导入行改成“跳过导入”。

### 6.3 提交幂等性

草稿提交后标记为 `COMPLETED`，再次提交同一草稿不会重复写入。

批量写入时仍应依赖 `source + sourceFingerprint` 的唯一性或事务内检查，防止重复请求造成重复交易。

## 7. 页面和交互设计

### 7.1 入口

在 `记录` 页顶部增加“导入”入口，跳转到独立导入页面，例如 `/records/import`。

不新增底部导航入口，避免主导航复杂化。

### 7.2 上传状态

用户选择随手记 `.xlsx` 文件并提交。成功后生成草稿，页面显示：

- 总行数
- 可导入数
- 同源重复跳过数
- 需要补映射的分类数
- 疑似重复数
- 不可导入数

文件不是 `.xlsx` 或表头不符合随手记结构时，不生成草稿并显示错误。

### 7.3 映射和重复确认状态

分类映射区域：

- 按 `交易类型 + 一级分类 + 二级分类` 聚合。
- 每个组合选择一个现有 BillBoard 分类。
- 已有家庭共享映射自动选中。
- 新映射在最终导入时保存。
- 有未完成映射时禁止确认导入。

疑似重复区域：

- 集中列出跨来源或手动同日同金额候选。
- 默认“保留导入”。
- 用户可以将个别行改为“跳过导入”。

不可导入区域：

- 显示前若干条原因。
- 不阻塞其它有效行导入。

### 7.4 完成状态

确认导入后批量写入 `Transaction`，完成页显示：

- 新增记录数
- 同源自动跳过数
- 用户选择跳过数
- 不可导入数

提供返回 `记录` 页的入口。

## 8. 服务端边界

建议模块边界：

- `src/lib/imports/sources/sui-shou-ji.ts`：随手记来源解析器。
- `src/lib/imports/fingerprint.ts`：来源指纹生成。
- `src/lib/imports/member-mapping.ts`：随手记成员名到家庭成员的映射。
- `src/lib/imports/category-mapping.ts`：分类映射 key 和已有映射查询。
- `src/lib/imports/drafts.ts`：草稿创建、读取、决策更新、确认导入。
- `src/app/(app)/records/import/page.tsx`：导入页面。
- `src/app/(app)/records/import/actions.ts`：上传、保存映射、确认导入 server actions。

实现时应保持来源解析器和导入草稿逻辑分离。后续支付宝、微信只新增来源解析器和对应字段规范化，不改导入确认主流程。

## 9. 错误处理

- 未登录：重定向登录页。
- 当前用户没有家庭成员身份：拒绝访问。
- 文件类型错误：显示“仅支持随手记 .xlsx 文件”。
- 文件超过 20MB 或可解析数据行超过 20000 行：显示文件过大错误。
- 表头不匹配：显示“无法识别随手记导出格式”。
- 金额、日期、币种等单行错误：标记该行为不可导入。
- 分类未映射：阻止确认导入，并提示需要补全映射。
- 草稿已完成：不重复提交，显示完成摘要。

## 11. 一期不做

- 不支持支付宝和微信解析。
- 不新增分类管理功能。
- 不自动创建分类。
- 不长期保存原始 Excel 文件。
- 不导入账户余额。
- 不把账户、商家、项目建成独立字段。
- 不提供复杂草稿管理列表。
- 不做强规则跨来源自动去重。
- 不支持多币种换算。

## 13. 设计自检

- 本设计只覆盖随手记一期导入，并预留多来源边界。
- 分类策略不新增 BillBoard 分类，符合现有 seed 管理约束。
- 金额、时间、软删除、家庭访问控制规则与现有业务约束一致。
- 同源重复和跨来源疑似重复使用不同处理方式，避免误跳过真实消费。
- 草稿落库解决大文件和多步确认流程的稳定性问题。
- 没有引入账户余额、转账、预算、附件等无关功能。
