# BillBoard UI i18n Design

日期：2026-04-30

## 1. 背景与目标

本设计定义 `BillBoard` 的中文 UI 支持与一键中英文切换。目标是让两人家庭记账应用默认以中文呈现，同时保留英文界面，且不增加记账阻力、不改变现有数据模型和业务流程。

本次改动继续服务三个核心体验：

- 快速记录家庭收入和支出。
- 按家庭成员视角查看记录。
- 按时间范围复盘家庭收支。

已确认范围：

- 默认语言为中文。
- 支持 `zh-CN` 与 `en-US` 一键切换。
- 语言偏好保存在当前浏览器。
- 中文支持仅限 UI 显示层。
- 分类名称只做显示映射，不修改数据库分类数据。

## 2. 假设

- 一键切换允许刷新当前页面一次，以保证 server components 和 client components 使用同一种语言重新渲染。
- 语言偏好不做账号级同步，也不跨浏览器同步。
- URL 继续只承载现有筛选和 drill-down 状态，语言不进入 URL。
- 数据库中的分类名、成员名、交易 note 保持原样显示或提交；仅内置分类名做 UI 显示翻译。
- 金额仍以整数分 `amountFen` 处理；日期和范围计算仍按 `Asia/Shanghai`。
- 英文 UI 继续保留，作为用户可切换的界面语言。

## 3. 已比较方案

### 3.1 Cookie + 轻量词典

服务端读取语言 cookie，缺省返回 `zh-CN`。页面和组件通过集中词典渲染文案。切换按钮写入 cookie 后刷新当前路由。

优点：

- 默认中文和刷新后的语言状态都稳定。
- server components 与 client components 能保持一致。
- 不污染现有筛选 URL。
- 不需要改数据库或认证模型。

缺点：

- 切换语言时会刷新当前页面一次。

结论：选择此方案。

### 3.2 LocalStorage + 客户端词典

语言保存在 `localStorage`，客户端组件读取后切换。

优点：

- 对纯客户端组件实现简单。
- 不需要服务端读取 cookie。

缺点：

- 现有大量页面文案在 server components 中，覆盖不完整。
- 若为切换语言把页面改成 client component，会增加复杂度和回归风险。

### 3.3 URL 参数 `?lang=zh-CN`

语言作为 query string 在导航间传递。

优点：

- 服务端天然知道当前语言。

缺点：

- 会污染现有 `range`、`perspective`、`type`、`category`、`record` 等筛选 URL。
- 导航、分享和 drill-down 链接会变复杂。
- 不符合当前应用简单直接的产品目标。

## 4. 语言状态设计

支持语言：

- `zh-CN`
- `en-US`

默认语言：

- cookie 缺失或非法时使用 `zh-CN`。

状态存储：

- 使用浏览器 cookie 保存语言。
- cookie 值只接受 `zh-CN` 和 `en-US`。
- 非法值按 `zh-CN` 处理。

切换行为：

- `LanguageToggle` 是小型 client component。
- 中文界面显示目标语言 `EN`。
- 英文界面显示目标语言 `中文`。
- 点击后写入 cookie，并刷新当前 route。
- 当前 URL 中的筛选参数保持不变。

入口位置：

- 登录页标题区右上角。
- 桌面端侧边栏品牌区。
- 移动端底部导航版本号行。

## 5. 词典与显示层设计

新增轻量 i18n 模块，集中管理：

- `locale` 类型和解析函数。
- UI 文案词典。
- 内置分类显示名映射。
- 日期、数字格式化 locale。

词典覆盖：

- 登录页标题、说明、输入框 label、按钮、错误提示。
- 导航：`Home`、`Add`、`Records`、产品说明和版本展示周边文案。
- Home：页面标题、说明、视角、范围、汇总卡片、趋势、分类、近期记录、空状态、交易数量。
- Add：页面标题、表单 label、类型、按钮、保存中状态、保存成功提示、后续操作。
- Records：页面标题、筛选、列表元信息、空状态、编辑抽屉、删除确认。
- 通用交易类型：收入、支出、全部类型。
- 通用元信息：记账人、创建人、无备注等。

分类名称：

- 数据库分类名保持英文。
- 页面显示时将已知内置分类映射为中文，例如 `Dining` 显示为“餐饮”、`Groceries` 显示为“买菜”、`Salary` 显示为“工资”。
- 未知分类名原样显示，避免新增复杂配置和不可达错误处理。
- 表单提交、筛选和查询仍使用 category id，不依赖显示名。

## 6. 组件与数据流

服务端流程：

1. 页面或 layout 读取 cookie。
2. `getLocale()` 解析语言，缺省为 `zh-CN`。
3. `getMessages(locale)` 返回词典。
4. 页面将 `messages` 或必要的 label props 传给组件。

客户端流程：

1. `LanguageToggle` 接收当前 `locale`。
2. 点击后写 cookie。
3. 使用 `router.refresh()` 让当前页面按新语言重新渲染。

组件边界：

- `AppShell` 负责把全局导航需要的文案传给 `DesktopNav` 和 `BottomNav`。
- 页面组件负责把页面文案传给表单、筛选、列表、图表和抽屉组件。
- `AppFiltersProvider` 继续只管理 `range` 和 `perspective`，不承载语言状态。
- `buildAppHref` 继续只保留共享筛选参数，不新增语言参数。

## 7. 格式化规则

金额：

- 继续使用 `amountFen` 和现有金额格式化策略。
- 中文和英文都保持两位小数。

数字：

- 中文界面使用 `zh-CN`。
- 英文界面使用 `en-US`。

日期：

- 日期显示使用当前 UI locale。
- 时区固定 `Asia/Shanghai`。
- 日期范围计算逻辑不变。

## 8. 错误处理

- 语言 cookie 缺失或非法时回落中文。
- 浏览器禁用 cookie 时，不新增复杂 fallback；语言偏好无法持久化属于可接受限制。
- 可控 UI 错误提示进入词典，例如登录失败、保存失败、更新失败、删除失败。
- 底层业务错误的英文文本可以保留用于调试，但页面优先展示可控文案。
- 未知分类名原样显示。

## 9. 实现边界

计划修改范围：

- `src/lib` 下新增轻量 i18n helper。
- `src/components` 中涉及文案的展示组件。
- `src/app` 中登录页、app layout、Home、Add、Records 页面。
- 相关 unit、integration、e2e 测试断言。

计划不修改：

- Prisma schema 和 migrations。
- seed 中的分类名称。
- 交易创建、更新、删除的数据结构和权限逻辑。
- 报表聚合、记录查询、时间范围计算逻辑。
- 认证 session 结构。
- URL 参数语义。
- 新增设置页或账号偏好表。

## 10. 测试计划

最小充分验证：

1. 运行 `npm run lint`。
2. 运行 `npm run test:unit`，覆盖 locale 默认值、cookie 解析、分类显示映射、导航文案渲染。
3. 运行 `npm run test:integration`，确认 server action 和查询逻辑不因 UI i18n 改动受影响。
4. 运行 `npm run test:e2e`，覆盖默认中文登录、切换英文、导航保留筛选参数、创建记录成功提示随语言变化。

如果 PostgreSQL、seed 数据或 Playwright 环境不可用，必须记录具体失败原因；提交前仍以仓库规则要求的完整验证为准。

## 11. 风险与应对

- 风险：默认中文导致现有测试按英文 label 查找失败。
  应对：同步更新测试断言，并保留英文切换的 e2e 覆盖。
- 风险：server component 与 client component 语言不一致。
  应对：语言来源统一为 cookie，切换后刷新当前 route。
- 风险：分类显示翻译误用于查询。
  应对：查询和提交继续使用 category id，翻译只发生在展示层。
- 风险：中文文案变长导致移动端按钮或导航溢出。
  应对：复用现有移动优先约束，并在 e2e 中覆盖关键导航和表单路径。
- 风险：语言参数污染现有筛选 URL。
  应对：语言不进入 URL，现有 `range` 和 `perspective` 保持原语义。

## 12. 设计结论

本次 i18n 采用 `Cookie + 轻量词典`。它在不改变业务数据、权限、查询和 URL 语义的前提下，让 `BillBoard` 默认中文并支持一键切换英文。实现重点是把 UI 文案集中、让服务端和客户端共享同一个语言来源，并保持快速记账与低配置成本。
