# 微信支付导入功能设计

日期：2026-05-07

## 1. 背景和目标

BillBoard 已支持随手记 `.xlsx` 导入，并已有导入草稿、分类映射记忆、同源去重、疑似重复确认和批量确认写入流程。微信支付导入的目标是在不增加记账阻力的前提下，把微信支付账单流水导入同一套家庭账本。

本期只支持微信支付导出的 `.xlsx` 账单流水文件。导入入口继续放在 `/records/import`，但上传前需要用户显式选择来源：`随手记` 或 `微信支付`。

## 2. 已确认假设

- 微信支付一期只支持 `.xlsx`。
- 微信支付导入复用现有 `ImportDraft`、`ImportDraftRow`、`ImportCategoryMapping` 和 `Transaction` 来源字段。
- 不新增 Prisma 模型，不新增 `ImportDraft` 字段。
- 微信支付来源标识使用 `wechat_pay`。
- 微信支付分类映射按 `收/支 + 交易类型 + 交易对方` 聚合。
- 微信支付账单的成员归属由页面下拉框决定。
- 成员下拉框默认值从文件头 `微信昵称` 推断；无法推断时使用当前登录用户。
- 修改微信账单归属成员时，直接批量更新该草稿所有未完成、非 `INVALID`、非 `SOURCE_DUPLICATE` 的微信行 `actorMemberId` 和 `createdByMemberId`。
- 微信账单本身没有独立的“记账人”和“成员”，所以同一个成员同时作为 `actorMemberId` 和 `createdByMemberId`。
- `中性交易` 标记为不可导入，不阻塞其它行。
- 退款、关闭、失败等状态标记为不可导入，不阻塞其它行。
- 未命中退款、关闭、失败关键词的未知状态按正常收入或支出导入。
- 金额继续使用整数分 `amountFen`。
- 时间继续按 `Asia/Shanghai` 语义解析，落库保存 UTC `Date`。
- 有效记录查询继续排除软删除数据。
- 所有访问和写入继续基于当前 session 的 `householdId` 和 `memberId` 校验。

## 3. 方案选择

采用方案：复用现有导入草稿主流程，新增微信支付来源解析器，并把上传页改成显式来源选择。

选择理由：

- 现有草稿流程已经覆盖大文件解析后的分步确认。
- 分类映射、同源去重、疑似重复、权限校验和批量确认可以复用。
- 微信支付只需要新增来源适配器和少量来源分发逻辑。
- 上传前选择来源可以给出更明确的格式错误提示。
- 不做完整多来源框架重构，避免把本次需求扩大。

不采用方案：

- 微信专用导入流程：会复制草稿、映射、去重和确认逻辑，维护成本高。
- 先重构完整多来源框架：结构更整齐，但超出本次微信支付导入需求。

## 4. 来源和数据模型

### 4.1 `ImportSource`

将现有单一来源扩展为两个来源：

- `sui_shou_ji`
- `wechat_pay`

需要把当前写死随手记来源的草稿服务通用化到“根据来源选择解析器和来源标签”。通用化只覆盖本次两个来源需要的边界，不新增插件式注册机制。

### 4.2 现有表复用

不新增数据库字段。微信支付导入继续使用：

- `ImportDraft.source = wechat_pay`
- `ImportDraftRow.mappingKey`
- `ImportDraftRow.primaryCategory`
- `ImportDraftRow.secondaryCategory`
- `ImportDraftRow.actorMemberId`
- `ImportDraftRow.createdByMemberId`
- `Transaction.source = wechat_pay`
- `Transaction.sourceFingerprint`
- `Transaction.sourceImportedAt`
- `ImportCategoryMapping.source = wechat_pay`

微信支付的 `primaryCategory` 保存微信 `交易类型`，`secondaryCategory` 保存微信 `交易对方`。页面仍显示为：

```text
支出 · 扫二维码付款 / 阿泉食杂店
```

## 5. 微信支付解析规则

### 5.1 工作簿识别

微信支付账单文件通常只有一个工作表，前若干行是说明和汇总信息，明细表头不固定在第 1 行。解析器需要在工作表中查找包含以下必需列的表头行：

- `交易时间`
- `交易类型`
- `交易对方`
- `商品`
- `收/支`
- `金额(元)`
- `支付方式`
- `当前状态`
- `交易单号`
- `商户单号`
- `备注`

如果无法找到这些表头，上传失败，不生成草稿，并显示微信支付格式识别错误。

### 5.2 文件头昵称

解析器读取形如 `微信昵称：[李环宇]` 的文件头信息，输出可选的 `ownerName`。草稿创建时用现有成员别名规则推断默认成员：

- `李环宇` 或 `老公` 默认选“老公”
- `晶晶` 或 `老婆` 默认选“老婆”
- 无法推断时默认选当前登录用户

该默认成员会写入微信草稿行的 `actorMemberId` 和 `createdByMemberId`。预览页允许用户修改。

### 5.3 字段转换

- `收/支=支出` 映射为 `EXPENSE`。
- `收/支=收入` 映射为 `INCOME`。
- `收/支=中性交易` 标记为 `INVALID`。
- 其它 `收/支` 值标记为 `INVALID`。
- `当前状态` 包含退款、关闭、失败等关键词时标记为 `INVALID`。
- 未命中这些异常关键词的状态按正常记录导入。
- `金额(元)` 转为整数分，必须大于 0。
- `交易时间` 按 `UTC+08:00` 解析为 `occurredAt`。
- `交易类型` 和 `交易对方` 去除首尾空白后作为分类映射 key。

### 5.4 备注生成

导入后写入 `Transaction.note`，保留微信关键原始信息，不把完整原始行塞入备注。

推荐格式：

```text
微信支付：交易类型；交易对方：交易对方；商品：商品；支付方式：支付方式；状态：当前状态；交易单号：交易单号；商户单号：商户单号；备注：备注
```

空字段不写入对应片段。微信原始备注为 `/` 时视为空。

### 5.5 来源指纹

微信支付同源精确去重基于稳定原始字段生成：

- 来源
- `收/支`
- `交易时间`
- `金额(元)`
- `交易类型`
- `交易对方`
- `商品`
- `支付方式`
- `当前状态`
- `交易单号`
- `商户单号`
- `备注`

同一家庭下已存在同 `source + sourceFingerprint` 的交易时，草稿行标记为 `SOURCE_DUPLICATE` 并自动跳过。该规则包含软删除交易，避免删除后重复导入。

## 6. 页面和交互

### 6.1 上传入口

`/records/import` 上传区新增来源选择：

- `随手记`
- `微信支付`

选择随手记时沿用现有解析和页面行为。选择微信支付时使用微信支付解析器。文件仍限制为 `.xlsx`，大小限制沿用现有 20MB。

错误提示按来源区分：

- 随手记：无法识别随手记导出格式。
- 微信支付：无法识别微信支付账单格式。

### 6.2 微信账单成员选择

微信草稿预览显示“账单归属成员”下拉框：

- 默认选中文件头昵称推断出的成员。
- 无法推断时默认选当前登录用户。
- 用户可以在确认导入前修改。
- 保存后批量更新该草稿所有未完成、非 `INVALID`、非 `SOURCE_DUPLICATE` 的微信行 `actorMemberId` 和 `createdByMemberId`。
- 已完成草稿不能修改成员。
- 随手记导入不显示该下拉框，继续沿用现有成员映射规则。

页面从草稿行中读取当前成员值展示，不新增草稿级数据库字段。

### 6.3 分类映射

分类映射 UI 沿用现有结构。微信支付映射项按 `收/支 + 交易类型 + 交易对方` 聚合。

映射保存后写入 `ImportCategoryMapping`，同一家庭、同一来源、同一收支类型、同一交易类型、同一交易对方下次自动复用。

### 6.4 重复处理

同源精确重复继续自动跳过。

跨来源或手动疑似重复继续使用现有规则：

- 同一家庭。
- 同一天。
- 同金额。
- 来源不同，或已有记录为手动记录。

疑似重复默认保留，用户可以改为跳过。

## 7. 服务端边界

建议模块边界：

- `src/lib/imports/types.ts`：扩展 `ImportSource`，增加微信支付解析输出所需的可选文件归属信息。
- `src/lib/imports/sources/wechat-pay.ts`：微信支付来源解析器。
- `src/lib/imports/sources/sui-shou-ji.ts`：保留现有随手记解析器。
- `src/lib/imports/drafts.ts`：把创建草稿、查询同源重复、确认写入中的来源常量改为来源参数；新增微信成员批量更新函数。
- `src/app/(app)/records/import/actions.ts`：上传 action 读取来源选择；新增保存微信账单归属成员 action。
- `src/app/(app)/records/import/page.tsx`：上传来源选择、微信成员下拉、来源化文案。
- `src/lib/i18n.ts`：补齐中英文文案。

实现时保持外科手术式修改，只抽取支持 `sui_shou_ji` 和 `wechat_pay` 必需的来源分发，不引入无关导入框架。

## 8. 错误处理

- 未登录：重定向登录页。
- 当前用户没有家庭成员身份：拒绝访问。
- 文件类型错误：提示仅支持 `.xlsx`。
- 文件超过 20MB 或可解析数据行超过 20000 行：显示文件过大错误。
- 来源未选择：回到导入页并提示选择导入来源。
- 微信表头不匹配：显示“无法识别微信支付账单格式”。
- 金额、时间、收支类型等单行错误：标记该行为不可导入。
- 中性交易：标记该行为不可导入。
- 退款、关闭、失败状态：标记该行为不可导入。
- 分类未映射：阻止确认导入，并提示需要补全映射。
- 草稿已完成：不重复提交，不允许修改成员。

## 9. 测试策略

### 9.1 单元测试

新增微信支付解析器测试，覆盖：

- 表头行查找。
- 文件头昵称解析。
- 默认成员推断需要的原始昵称输出。
- `收/支` 到 `TransactionType` 的转换。
- 中性交易不可导入。
- 退款、关闭、失败状态不可导入。
- 未知但非异常状态可导入。
- 金额转整数分。
- `UTC+08:00` 时间解析。
- 分类映射 key。
- 备注生成。
- 来源指纹稳定性。
- 无法识别工作簿时报错。

### 9.2 集成测试

覆盖：

- 微信支付上传生成草稿。
- 默认成员从文件头昵称推断。
- 无法推断昵称时回退当前登录用户。
- 保存微信账单归属成员后批量更新草稿行。
- 已完成草稿不能修改成员。
- 微信支付分类映射保存和复用。
- 确认导入写入 `wechat_pay` 来源交易。
- 重复导入同一微信行自动跳过。
- 微信导入与手动记录或其它来源同日同金额时标记疑似重复。
- 家庭权限隔离。

### 9.3 E2E 测试

新增微信支付导入路径：

1. 登录。
2. 从记录页进入导入页。
3. 选择微信支付。
4. 上传小型微信支付测试文件。
5. 查看或调整账单归属成员。
6. 保存分类映射。
7. 确认导入。
8. 返回记录页看到导入记录。

## 10. 一期不做

- 不支持微信支付 `.csv`。
- 不支持加密文件解密。
- 不把支付方式、交易对方、交易单号建成独立业务字段。
- 不自动创建 BillBoard 分类。
- 不导入中性交易。
- 不做退款净额抵消。
- 不新增独立微信导入页面。
- 不新增复杂草稿管理列表。
- 不提供多来源插件注册系统。

## 11. 验证要求

实现完成后按仓库规则执行完整验证：

```text
npm run lint

npm run test:integration
npm run test:e2e
```

涉及 server actions、导入解析、认证、数据库写入和端到端导入流程，因此完整验证全部通过后才能提交实现。

## 12. 设计自检

- 本设计聚焦微信支付 `.xlsx` 导入，没有引入无关页面或分类管理能力。
- 微信支付复用现有导入草稿链路，避免复制确认、去重和权限逻辑。
- 成员下拉不新增数据库字段，符合已确认约束。
- 分类映射粒度明确为 `收/支 + 交易类型 + 交易对方`。
- 中性交易和退款类状态不会污染收入支出账本。
- 同源去重和跨来源疑似重复继续沿用现有语义。
- i18n、测试和验证要求覆盖了新增用户可见行为。
